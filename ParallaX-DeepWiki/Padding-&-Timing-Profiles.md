# Padding & Timing Profiles
Relevant source files

- [src/protocol/data.rs](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/protocol/data.rs)
- [src/traffic.rs](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs)

Traffic obfuscation in ParallaX is primarily handled by the `PaddingProfile` and `TimingProfile` structures. These profiles implement statistical shaping of packet sizes and inter-packet arrival times to mimic the characteristics of legitimate HTTPS traffic and resist traffic analysis (fingerprinting).

## Padding Profiles

The `PaddingProfile`[src/traffic.rs#19-22](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L19-L22) manages the addition of cryptographic padding to data records. Unlike static padding, ParallaX uses a weighted distribution model to target common Maximum Transmission Units (MTUs) and packet sizes observed in typical web traffic.

### Padding Distribution Logic

When `PaddingProfile::apply` is called [src/traffic.rs#54-67](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L54-L67) the system calculates a padding length based on the following priority:

1. Common MTU Targeting (55% weight): In 55% of cases, the system attempts to pad the packet so that the final size matches one of the `OBSERVED_PACKET_TARGETS`[src/traffic.rs#36-38](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L36-L38) These targets include common values like 64, 569, 1353, and 1500 bytes. The calculation subtracts the payload length and the 2-byte overhead from the target [src/traffic.rs#77-84](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L77-L84)
2. Bucketed Randomness (45% weight): If a target is not chosen, the system uses a bucketed range to select a random padding length:

- Small (70% of remaining): Capped at 64 bytes [src/traffic.rs#88-89](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L88-L89)
- Medium (22% of remaining): Capped at 256 bytes [src/traffic.rs#90-91](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L90-L91)
- Large (8% of remaining): Up to the configured `max_padding`[src/traffic.rs#92-93](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L92-L93)

### Wire Format & Lifecycle

Padding is integrated into the `DataRecordCodec` lifecycle. The padding is applied *before* AEAD encryption to ensure the padding itself is encrypted and indistinguishable from data.

| Component | Description |
| --- | --- |
| Payload | The actual application data. |
| Random Padding | Bytes generated by `rng.fill_bytes`[src/traffic.rs#64](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L64-L64) |
| Length Field | A 2-byte Big-Endian field indicating the length of the padding [src/traffic.rs#65](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L65-L65) |

## Timing Profiles

The `TimingProfile`[src/traffic.rs#25-28](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L25-L28) introduces artificial delays between packet transmissions to disrupt timing-based analysis of the protocol.

### Delay Distribution

Similar to padding, the `sample_delay` function [src/traffic.rs#121-136](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L121-L136) uses a weighted approach:

- Observed Delays (60% weight): Samples from `OBSERVED_DELAY_MS`[src/traffic.rs#40](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L40-L40) which contains values like 0ms, 3ms, 25ms, 410ms, and 804ms. This mimics the jitter found in real-world network conditions [src/traffic.rs#128-132](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L128-L132)
- Linear Random (40% weight): Samples uniformly between the configured `min_delay_ms` and `max_delay_ms`[src/traffic.rs#133-135](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L133-L135)

## Integration with DataRecordCodec

The `DataRecordCodec` acts as the orchestrator for traffic shaping. It holds the `PaddingProfile` and ensures that every outgoing record is shaped according to the profile before being wrapped in a TLS Application Data record [src/protocol/data.rs#41](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/protocol/data.rs#L41-L41)

### Maximum Plaintext Calculation

Because TLS records have a fixed maximum size (`MAX_TLS_RECORD_PAYLOAD`), the addition of padding and AEAD tags reduces the available space for actual application data. The function `max_plaintext_len` calculates this limit:

`max_plaintext_len = MAX_TLS_RECORD_PAYLOAD - (max_padding + AEAD_TAG_LEN + PADDING_LEN_FIELD)`

This ensures that even with the maximum possible padding, the resulting encrypted record does not exceed the TLS limit [src/protocol/data.rs#65-68](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/protocol/data.rs#L65-L68)

## Error Handling

The traffic subsystem defines specific errors for padding validation in `TrafficError`[src/traffic.rs#8-16](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L8-L16):

- `InvalidPaddingRange`: Thrown if `min > max` during profile creation [src/traffic.rs#11](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L11-L11)
- `PaddedFrameTooShort`: Thrown if `remove` is called on a buffer smaller than the 2-byte length field [src/traffic.rs#13](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L13-L13)
- `PaddingLengthOutOfRange`: Thrown if the length field indicates more padding than the total buffer size [src/traffic.rs#15](https://github.com/yuzeguitarist/ParallaX/blob/77045cea/src/traffic.rs#L15-L15)
